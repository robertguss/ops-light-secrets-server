### jdx's reply (landed 2026-07-15, after T03 closed)

Verbatim from the thread (comment notes it was generated by Codex, posted
from jdx's account):

> Thanks for asking first.
>
> A direct-HTTP implementation for the existing `vault` provider would be
> welcome. I would prefer replacing the CLI-backed internals while preserving
> the current configuration, rather than introducing a separate provider. It
> should retain the existing authentication and TLS behavior and include
> compatibility tests for both Vault and OpenBao.
>
> I may also be interested in a fnox-native server protocol, but I would need
> to think through the design and long-term implications quite a bit more. A
> design discussion would be welcome, especially if it starts with concrete
> use cases, desired capabilities, and what a native protocol would enable
> beyond the existing provider interface or Vault-compatible approach. There
> is no commitment that fnox would adopt it yet, but I am open to exploring
> it.
